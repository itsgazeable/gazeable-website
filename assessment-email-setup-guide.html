Course Module — Email Delivery Setup

How to Add "Email My Results" to Any Quiz, Form, or Tool

The Cloudflare + Resend Method — free, reliable, works for any client on any platform.

What You Will Build

A working "send my results" button that emails the person who filled out your form or quiz — and automatically sends you a copy too.

1. What You'll Build

By the end of this module, you'll have a permanent, reusable system that any form or quiz can call to send a personalized email — to the person who filled it in, with a copy to you.

The flow, in plain terms

  • Someone fills out a quiz or form on your website
  • They type in their email and click "Send My Results"
  • Their browser sends a request to a small piece of code you control (a Cloudflare Worker)
  • That code asks a proper email service (Resend) to deliver the email
  • They receive their results. You receive a copy automatically.
Why This Matters

Most free "email from a form" services only let you send to yourself — not to whoever filled in the form. This module solves that specific problem, for free, in a way that works for any client project.

2. Why This Method (And Not the Others)

There are several popular "no backend needed" form services. We tested the most common ones before landing on this approach. Here's why each one falls short for this specific use case — sending a result back to the person who submitted the form, not just to you.

EmailJS

Built specifically to send email from browser JavaScript. Sounds perfect — until you discover the free plan only sends to the email address tied to your connected account. Sending to a different person's address requires a paid plan, and even then requires manually allow-listing your domain.

Web3Forms

A simple, free form-to-email API. Same root limitation as EmailJS — submissions only deliver to the account owner's registered email. You'll receive the data, but the person who filled out the form will never get their results.

Formsubmit.co

No-signup form backend. Reliable for collecting submissions to yourself. The dynamic recipient override fields are documented but don't function as a true "send to anyone" feature on the free tier — the same spam-prevention restriction applies.

Resend, called directly from the browser

Resend actually can send to any address from your own domain — exactly what we need. But calling its API directly from a browser triggers a CORS error. Browsers block this on purpose: API keys should never be exposed in code anyone can view by right-clicking "View Source."

The Insight

Every free browser-to-email service restricts who can receive the email, to prevent spam abuse. Resend doesn't have that restriction — because it's designed to be called from a server, not directly from a browser. The fix isn't a different email service. It's adding a server in between.

Comparison at a glance

Requirement EmailJS Web3Forms Formsubmit Cloudflare + Resend
Send to any recipient ✕ Blocked ✕ Blocked ✕ Blocked ✓ Yes
Send from your own domain Paid only N/A N/A ✓ Yes
No CORS errors ✓ Yes ✓ Yes ✓ Yes ✓ Yes (via Worker)
CC yourself automatically Partial ✕ No Partial ✓ Yes
API key hidden from public ✕ Exposed ✕ Exposed N/A ✓ Hidden in Worker
Free to run ✓ Yes ✓ Yes ✓ Yes ✓ Yes

That missing server is called a Cloudflare Worker — and it's free, takes about ten minutes to set up, and you'll reuse the exact same one for every future project.

3. Before You Start

Make sure you have these three things ready. None of them cost money at this stage.

  • A domain name you own (e.g. yourbusiness.com) — needed so you can send email "from" your own address
  • Access to your domain's DNS settings (wherever you bought the domain — GoDaddy, Namecheap, Cloudflare, your website host, etc.)
  • An email address at that domain to receive copies of submissions (e.g. hello@yourbusiness.com)
No Domain Yet?

You can still test this whole setup using Resend's shared sending domain (onboarding@resend.dev) with zero DNS setup. Skip straight to Part 2, and in Part 1 simply note your Resend API key — you won't need to verify a domain.

Part 1 — Set Up Resend

Resend is the email service that actually delivers the message. Think of it as the engine — it does the sending, but only when asked by something it trusts (your Worker, built in Part 2).

1
Create your Resend account
  1. Go to resend.com and sign up using the email address you want submissions copied to.
  2. Confirm your email address if asked.
2
Add and verify your sending domain
  1. In the Resend dashboard, go to Domains → Add Domain.
  2. Enter your domain (e.g. yourbusiness.com) — not a subdomain, the root domain.
  3. Resend will show you 2–3 DNS records to add (usually a TXT record for verification and one or two for DKIM/SPF — these prove you own the domain and prevent your emails from being marked as spam).
  4. Go to wherever you manage your domain's DNS — your domain registrar or hosting provider — and add each record exactly as Resend shows it.
  5. Return to Resend and click Verify. DNS changes can take anywhere from a few minutes to an hour to take effect.
  6. Once verified, you'll see a green checkmark next to your domain. Do not continue until you see this.
Where To Find DNS Settings

If your client doesn't know where their domain DNS is managed, check where they registered the domain (look at their original purchase email) or check their website host's dashboard — most hosting platforms have a DNS or Domain settings section.

3
Create your API key
  1. Go to API Keys → Create API Key.
  2. Name it something memorable, like "Client Tools" or the client's business name.
  3. Copy the key immediately — it starts with re_ and will only be shown once.
  4. Paste it somewhere safe temporarily — you'll need it in Part 2, Step 3.
Security Note

This API key must never appear in any public-facing HTML, JavaScript, or anything a website visitor's browser can see. It only ever goes inside the Cloudflare Worker code in Part 2, which runs on a server — never in the browser.

Part 2 — Set Up the Cloudflare Worker

The Worker is a small piece of code that sits between your form and Resend. It receives the request from the browser, adds your secret API key (which the browser never sees), and forwards everything to Resend.

1
Create a Cloudflare account
  1. Go to cloudflare.com and sign up. No credit card required for this.
2
Create your Worker
  1. In your Cloudflare dashboard, go to Workers & Pages.
  2. Click Create Application, then Create Worker.
  3. Choose "Start with Hello World!"
  4. Name your Worker — use something clear like clientname-email or yourbusiness-email. This name becomes part of your permanent Worker URL.
  5. Click Deploy. You'll see a default "Hello World" response — this is expected. You'll replace the code in the next step.
3
Replace the Worker code
  1. Click Edit Code on your newly deployed Worker.
  2. Delete everything in the editor.
  3. Paste the code below.
  4. Replace re_YOUR_API_KEY_HERE with the actual API key you copied in Part 1, Step 3.
  5. Replace grace@yourbusiness.com with the actual sending address on your verified domain.
  6. Click Save and Deploy.

The Worker code

export default {
  async fetch(request) {

    if (request.method === 'OPTIONS') {
      return new Response(null, {
        headers: {
          'Access-Control-Allow-Origin': '*',
          'Access-Control-Allow-Methods': 'POST, OPTIONS',
          'Access-Control-Allow-Headers': 'Content-Type',
        }
      });
    }

    if (request.method !== 'POST') {
      return new Response('Method not allowed', { status: 405 });
    }

    try {
      const body = await request.json();
      const { to, subject, text, html } = body;

      const payload = {
        from: 'Your Business <grace@yourbusiness.com>',
        to: [to],
        cc: ['grace@yourbusiness.com'],
        reply_to: to,
        subject: subject,
      };

      if (html) {
        payload.html = html;
      } else {
        payload.text = text;
      }

      const res = await fetch('https://api.resend.com/emails', {
        method: 'POST',
        headers: {
          'Authorization': 'Bearer re_YOUR_API_KEY_HERE',
          'Content-Type': 'application/json'
        },
        body: JSON.stringify(payload)
      });

      const data = await res.json();

      return new Response(JSON.stringify(data), {
        status: res.status,
        headers: {
          'Content-Type': 'application/json',
          'Access-Control-Allow-Origin': '*'
        }
      });

    } catch (err) {
      return new Response(JSON.stringify({ error: err.message }), {
        status: 500,
        headers: {
          'Content-Type': 'application/json',
          'Access-Control-Allow-Origin': '*'
        }
      });
    }
  }
};

What this code actually does

  • Waits for a request containing three things: who to send to (to), the subject line (subject), and the message body (text or html)
  • Builds an email "package" that's always from your business address, always CC's you, and sets reply-to as the form-filler's email so you can reply directly to them
  • Sends that package to Resend's API — server to server, so there's no browser restriction
  • Sends back a response so your form can show a success message
  • If anything goes wrong, it catches the error instead of crashing silently
4
Note your Worker's permanent URL
  1. After deploying, Cloudflare shows your Worker's URL. It will look like:
https://yourworkername.yourusername.workers.dev

This URL never changes. Save it — you'll use it in every form or quiz you connect to this system.

Part 3 — Connect Your Form or Quiz

Now that the Worker exists, any form or quiz just needs to send a request to it when the "Send My Results" button is clicked. This works the same way no matter what platform hosts your form — Systeme.io, a custom website, WordPress, anything that allows custom JavaScript.

1
Build the email content in JavaScript

Use template literal backticks (the ` character, not regular quotes) to build the email body. This lets you write multi-line HTML directly in your JavaScript without escaping every line break — and it's what makes the email actually look designed instead of plain text.

var emailBody = `
  <div style="font-family: Arial, sans-serif;">
    <h2>Your Results</h2>
    <p>Here's what we found...</p>
  </div>
`;
2
Send the request when the button is clicked
fetch('https://yourworkername.yourusername.workers.dev', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    to: email,        // the address the person typed in
    subject: 'Your Results',
    html: emailBody
  })
}).then(function() {
  // show a success message to the user
}).catch(function() {
  // show success anyway — don't block them on a network hiccup
});
Why "Show Success Anyway" In The Catch Block

If the email send fails due to a temporary network issue, you still want the person to feel like their submission worked — you'll still receive the data on your end via the CC. Never make the visitor feel like something broke when it likely didn't.

Part 4 — Test Your Setup

Before handing this off or going live, test the full path end to end.

  1. Open the live form or quiz in a browser.
  2. Fill it out completely using an email address you can check (your own, or a second test inbox).
  3. Click the "Send My Results" button.
  4. Check the test inbox — the email should arrive within a few seconds.
  5. Check your own inbox (the one CC'd in the Worker code) — you should receive a copy too.
  6. If using Gmail, check the Spam folder on the very first send — brand-new sending domains sometimes land there once before Gmail trusts them.

Troubleshooting Common Errors

Browser console shows a CORS error

This means your form is calling Resend directly instead of your Worker. Double check the fetch() URL in your form's code points to your workers.dev URL, not api.resend.com.

Browser console shows a 403 error

This is a domain restriction error — usually means you're accidentally still using EmailJS, Web3Forms, or Formsubmit code somewhere instead of the Worker. Search your form's code for any of those service names and remove leftover references.

Browser console shows a 422 error

The data being sent doesn't match what's expected. Check that your fetch() body includes exactly to, subject, and either text or html — no extra or missing fields.

Email never arrives, no error shown

Check that your domain shows a green Verified checkmark in the Resend dashboard. If it's still pending, sends will silently fail until DNS finishes propagating.

Email arrives to you (CC) but not to the form-filler

Double check the to field in your fetch() call is actually using the variable holding what the person typed — not a hardcoded email address left over from testing.

"Something went wrong" message shows even though the email actually sent

This is a cosmetic issue, not a delivery issue — it means your success/error logic is checking for a response format that doesn't match what's returned. Simplify your .then() block to show success on any non-error response, since the email is already confirmed working.

Reusing This Setup for Future Projects

This is the part that makes this method worth learning properly: once built, the exact same Worker can power unlimited forms and quizzes for the same business.

  • Each new form just needs its own fetch() call pointing at the same Worker URL
  • Change only the to, subject, and html/text content for each new use case
  • No new Cloudflare or Resend setup required for additional forms on the same domain
  • For a new client with a different domain, repeat Part 1 (their domain) and Part 2 (a new Worker) — the code itself barely changes
Client Delivery Tip

When setting this up for a client, create the Worker inside their own Cloudflare account (not yours) so they retain full ownership and control after the engagement ends. Use their Resend account and their domain throughout.
